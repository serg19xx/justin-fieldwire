# Руководство для заказчика: личный календарь FieldWire

**Аудитория:** заказчик, QA, менеджеры проектов.  
**Версия:** 1.0 (календарь v2 — `requires_presence`)  
**Сайт:** https://fieldwire.medicalcontractor.ca

---

## Важно: это не календарь задач

В разделе **Tasks** календарный вид задач **отключён** (остались список и Gantt).  
Новый **Calendar** — отдельный **личный календарь пользователя**: встречи, напоминания, визиты на объект. События **не связаны** с задачами проекта (`fw_prj_tasks`).

---

## Два режима календаря

| | **Глобальный календарь** | **Календарь проекта** |
|---|--------------------------|------------------------|
| **Где открыть** | Пункт **Calendar** в верхнем меню → `/calendar` | Проект → боковое меню **Calendar** (между Schedule и Photos) |
| **Что видно** | Все ваши события: личные + события из **всех** проектов | Личные события + события **текущего** проекта |
| **Что можно редактировать** | Только **личные** (без привязки к проекту) | Только события **этого проекта** |
| **События других проектов** | Видны, **только просмотр** (серые) | **Скрыты** |
| **Личные события** | Синие, **редактируемые** | Светло-синие, **только просмотр** |

**Правило:** личное событие создаётся и меняется в **глобальном** календаре. Событие проекта — во **вкладке Calendar** внутри нужного проекта.

---

## Типы событий

При создании/редактировании есть чекбокс **Requires my presence** («Требуется моё присутствие»).

| Тип | Чекбокс | Поведение |
|-----|---------|-----------|
| **Reminder** (напоминание) | выключен | Можно создавать **сколько угодно** событий на одно время. Конфликты **не проверяются**. |
| **Requires my presence** | включён | Система ищет **пересечения** с другими presence-событиями **этого же пользователя** (личный календарь + все проекты). При конфликте — предупреждение. |

### При конфликте presence

1. Первое нажатие **Save** — показывается список занятых слотов.  
2. **Save anyway** — сохранить несмотря на пересечение (осознанное решение пользователя).  
3. **Cancel** — отменить.

Напоминания (Reminder) **никогда** не блокируют сохранение и **не участвуют** в проверке чужих presence-событий.

---

## Внешний вид (легенда)

**Глобальный календарь**

- Синий — личное событие (можно редактировать)
- Серый — событие проекта (только просмотр)
- Янтарная рамка — **Requires my presence**

**Календарь проекта**

- Зелёный — событие проекта (можно редактировать)
- Светло-синий — личное (только просмотр)
- Янтарная рамка — **Requires my presence**

---

## Как создать событие

### Месяц (Month)

1. **Один клик** по дню — только выделение даты (диалог **не** открывается).  
2. **Двойной клик** по дню или кнопка **+ New event** — открывается форма.  
3. Для события с временем по умолчанию: **10:00–12:00** (можно изменить).

### Неделя / День (Week / Day)

- **Перетащите** мышью интервал времени — создаётся событие на выбранный диапазон.  
- Или **двойной клик** по ячейке времени.

### Поля формы

- **Title** — обязательно  
- **Requires my presence** — тип события  
- **Notes**, **Location** — по желанию  
- **All day** — целый день  
- **Start / End date** и **Start / End time** (если не весь день)

## Сценарии тестирования

### Тест 1 — Глобальный календарь, личное напоминание

1. Войти под пользователем.  
2. Меню **Calendar** → **+ New event**.  
3. Заголовок, дата/время, чекбокс **Requires my presence** — **выключен**.  
4. **Save**.  
5. **Ожидание:** событие синее, открывается на редактирование, удаляется.

### Тест 2 — Два напоминания на одно время

1. Создать два Reminder на **одинаковое** время.  
2. **Ожидание:** оба сохраняются без предупреждения.

### Тест 3 — Presence и конфликт

1. Создать событие A: **Requires my presence**, например Пн 10:00–12:00.  
2. Создать событие B на **пересекающееся** время, тоже с presence.  
3. **Ожидание:** список конфликтов, кнопка **Save anyway**.  
4. **Save anyway** — B сохраняется.  
5. **Cancel** на шаге 3 — B не сохраняется.

### Тест 4 — Кросс-проектный конфликт

1. В **проекте A** создать presence-событие на время T.  
2. В **глобальном** календаре создать presence на то же T.  
3. **Ожидание:** конфликт виден (событие из проекта A в списке).  
4. Reminder на то же T в глобальном календаре — **без** конфликта.

### Тест 5 — Календарь проекта

1. Открыть **проект** → **Calendar**.  
2. Создать событие проекта (зелёное).  
3. **Ожидание:** в глобальном календаре оно **серое**, только просмотр.  
4. Личное событие в проекте — светло-синее, только просмотр; редактировать в `/calendar`.

### Тест 6 — События другого проекта

1. Создать событие в **проекте B**.  
2. Открыть **Calendar** в **проекте A**.  
3. **Ожидание:** событие из B **не отображается**.  
4. В **глобальном** календаре событие B **видно** (серое).

### Тест 7 — Редактирование и удаление

1. Клик по своему редактируемому событию → изменить заголовок → **Save**.  
2. **Delete** в диалоге → подтвердить.  
3. Чужое (read-only) событие — подсказка «Read-only here», без Save/Delete.

## Ограничения (ожидания)

- Календарь **не синхронизируется** с Google/Outlook.  
- События **не создают** и **не меняют** задачи в Tasks.  
- Проверка presence — только для **текущего** залогиненного пользователя.  
- **Save anyway** — намеренный обход конфликта; система не запрещает двойное бронирование жёстко.

---

## Если что-то не работает

| Симптом | Что проверить |
|--------|----------------|
| Ошибка при сохранении (500) | Сообщить разработчику; на сервере должна быть таблица `fw_calendar_events` и колонка `requires_presence`. |
| Нет пункта Calendar | Обновить страницу (Ctrl+F5); нужна свежая сборка фронта. |
| Конфликт не показывается | У обоих событий должен быть включён **Requires my presence**; время должно пересекаться. |
| Не редактируется событие проекта | Редактировать в **Calendar** внутри того же проекта, не в глобальном. |

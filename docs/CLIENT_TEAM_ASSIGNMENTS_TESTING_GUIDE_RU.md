# Руководство для заказчика: назначение сотрудников на задачи и проверка по ролям

**Аудитория:** заказчик, QA, менеджеры проектов.  
**Версия:** 1.0  
**Сайт:** https://fieldwire.medicalcontractor.ca

**Веб (после деплоя):** https://fieldwire.medicalcontractor.ca/CLIENT_TEAM_ASSIGNMENTS_TESTING_GUIDE_RU.html  
**English:** https://fieldwire.medicalcontractor.ca/CLIENT_TEAM_ASSIGNMENTS_TESTING_GUIDE.html

---

## Что проверяем в этом документе

Пошаговая проверка **назначения людей на задачи проекта** и того, **что видит каждая роль** в веб-интерфейсе PM и в мобильном интерфейсе бригадира / работника.

**Не входит в это руководство** (заказчик уже умеет, в системе есть тестовые данные):

- приглашение и добавление людей в команду проекта;
- создание проектов;
- создание задач с нуля.

**Отдельно — недоделанная часть:** система **сообщений и push/email-уведомлений** о назначениях. Интерфейс чата по слотам расписания может быть виден, но **реальная доставка SMS, push и email по событиям назначения пока не считается готовой к приёмке**. Это описано в разделе 8.

---

## 1. Два уровня «команды»

| Уровень | Где задаётся | Зачем |
|---------|--------------|-------|
| **Команда проекта** | Проект → **Team** | Кто вообще допущен к проекту (общий состав). |
| **Назначение на задачу** | Проект → **Tasks** → карточка задачи | Кто **конкретно** работает на **этой** задаче. |

Для полевых ролей важен **второй уровень**: работник или бригадир увидит задачу в своём приложении только если он указан на задаче как **участник** или **бригадир задачи**.

---

## 2. Роли на задаче (назначение сотрудников)

Откройте проект → **Tasks** → выберите задачу (из списка или из **Gantt**).

В панели редактирования задачи:

### 2.1 Foreman / Brigadier (бригадир задачи)

- Поле на вкладке **General**: **Foreman / Brigadier**.
- Сохраняется как **task lead** (ответственный бригадир **на этой задаче**).
- Может обновлять прогресс и полевые данные по задаче, если он назначен lead.

### 2.2 Assigned Team Members (участники задачи)

- Вкладка **Team** → **+ Add Team Member**.
- Список людей из команды проекта (и доступных работников).
- Сохраняется как **team_members** — обычные исполнители на задаче.
- Видят задачу в своём приложении, но **не могут** менять прогресс (только просмотр), если они не task lead.

### 2.3 Как это отображается у PM в Team → Tasks

В режиме просмотра **Tasks** у сотрудника каждая строка помечена:

- **Task lead** — человек назначен бригадиром задачи;
- **Team member** — человек в составе задачи без роли lead.

---

## 3. Статусы задачи и где они видны

Система группирует статусы по-разному для **PM (расписание сотрудника)** и для **полевых ролей (вкладки в /tasks)**.

### 3.1 Статусы в базе (кратко)

| Статус задачи | Смысл для проверки |
|---------------|-------------------|
| `planned` | Запланировано, ещё не в работе |
| `scheduled` | Запланировано с привязкой к расписанию |
| `scheduled_accepted` | Запланировано, принято исполнителем |
| `in_progress` | В работе |
| `partially_completed` | Частично выполнено |
| `delayed_due_to_issue` | Задержка |
| `ready_for_inspection` | Готово к проверке |
| `completed` | Завершено (архив) |

### 3.2 PM: Team → **Tasks** → вкладки Planned | Current | Archive

| Вкладка PM | Какие статусы попадают | Что означает |
|------------|------------------------|--------------|
| **Planned** | `planned`, `scheduled`, `scheduled_accepted` | Планирование и подготовка — **полная картина для PM**, в том числе пока задача ещё «на бумаге». |
| **Current** | Все активные (`in_progress`, `partially_completed`, задержки, inspection и т.д.) | **Current** — одна «главная» активная задача; **Upcoming** — остальные активные по датам. |
| **Archive** | `completed` | Завершённые назначения. |

Режим **только просмотр**: PM здесь **не** добавляет и **не** снимает людей с задач — назначение делается в **Tasks** / **Gantt**.

### 3.3 Бригадир / работник: `/tasks/projects/{id}` → Planned | In progress | Closed

| Вкладка поля | Какие статусы | Примечание |
|--------------|---------------|------------|
| **Planned** | `planned`, `scheduled`, `scheduled_accepted` | Предстоящая работа **по уже назначенным** задачам |
| **In progress** | `in_progress`, `partially_completed`, `delayed_due_to_issue`, `ready_for_inspection` и др. активные | Текущая работа |
| **Closed** | `completed` | Закрытые задачи |

**Важно:** работник **не** видит задачи, на которые он **не назначен** (ни lead, ни member). Бригадир в режиме **All tasks** видит **все задачи проекта**; в **My tasks** — только где он участник или lead.

### 3.4 Сводная таблица «кто что видит»

| Статус задачи | PM Team → Tasks (Planned / Current / Archive) | Работник (если назначен) | Бригадир My tasks | Бригадир All tasks |
|---------------|-----------------------------------------------|--------------------------|-------------------|---------------------|
| planned | Planned | Planned | Planned | Planned |
| scheduled | Planned | Planned | Planned | Planned |
| scheduled_accepted | Planned | Planned | Planned | Planned |
| in_progress | Current | In progress | In progress | In progress |
| partially_completed | Current | In progress | In progress | In progress |
| ready_for_inspection | Current | In progress | In progress | In progress |
| delayed_due_to_issue | Current | In progress | In progress | In progress |
| completed | Archive | Closed | Closed | Closed |
| Не назначен на задачу | Не попадает в Tasks сотрудника | **Не видит** | **Не видит** (My) | **Видит** (All) |

## 4. Что видит PM: Gantt, Tasks, Team

### 4.1 Gantt (диаграмма в разделе Tasks)

**Где:** Проект → **Tasks** → переключатель вида **Gantt**.

**Зачем PM:**

- видеть **сроки всех задач** проекта на временной шкале;
- понимать **пересечения и зависимости**;
- быстро открыть задачу и **назначить бригадира и команду** через ту же панель редактирования.

**Gantt не заменяет Team:** на полосках Gantt **нет** персонального «расписания работника» — там **задачи проекта**. Кто на какой задаче — задаётся в карточке задачи; сводка **по человеку** — в **Team → Tasks**.

### 4.2 Tasks (список + карточка задачи)

**Где:** Проект → **Tasks** → список или Gantt → клик по задаче.

**Зачем PM:** единственное место **назначения** Foreman / Team members и смены статуса задачи.

### 4.3 Team — глобальный и проектный

| Место | Путь | Что показывает |
|-------|------|----------------|
| **Глобальный Team** | Верхнее меню → **Team** | Все работники / подрядчики по доступным проектам (та же таблица, что в проекте). |
| **Team проекта** | Проект → боковое меню **Team** | Только люди, привязанные к **этому** проекту. |

**Кнопки в строке сотрудника:**

| Кнопка | Назначение |
|--------|------------|
| **Details** | Профиль: контакты, профессия, языки, экстренный контакт (**без** списка проектов). |
| **Tasks** | **Расписание задач сотрудника (view only)** — вкладки Planned / Current / Archive. |

**Глобальный Team → Tasks:** если у человека назначения в **нескольких** проектах — проекты в **аккордеоне** (раскрыть нужный проект).  
**Team проекта → Tasks:** сразу три вкладки **без** аккордеона — только этот проект.

**Зачем два Team:** глобальный — «кто где занят по всей компании»; проектный — быстрая проверка **только этого объекта**.

## 5. Что видит бригадир (мобильный / task-layout)

**Вход:** учётная запись с ролью **Foreman** (бригадир).  
**Главный экран:** дашборд с обзором проектов, открытых задач, размера команды.  
**Маршруты:** меню **Tasks** → **My projects** → проект.

### 5.1 Дашборд бригадира

- **Active projects**, **Open tasks**, **Team (unique)** — агрегаты по доступным проектам.
- Блок **Recent projects** → ссылка **View tasks** в список проектов.

### 5.2 Список задач в проекте

Переключатель (только у бригадира):

| Режим | Содержимое |
|-------|------------|
| **My tasks** | Задачи, где он **task lead** или **team member** |
| **All tasks** | **Все** задачи проекта (контроль бригадира на объекте) |

Вкладки статуса: **Planned | In progress | Closed** (см. таблицу в §3).

### 5.3 Права на задаче

- Если бригадир — **task lead** на задаче → может менять **прогресс** и полевые поля.
- Если только **team member** → задача **только для просмотра** (как у работника).

---

## 6. Что видит работник (мобильный / task-layout)

**Вход:** учётная запись **Worker** или **Contractor**.  
**Главный экран:** **Your projects** — проекты, где есть назначения на задачи.

### 6.1 Навигация

1. **Tasks** → **My projects** (вкладки **Current project** / **Planned** по жизненному циклу **проектов**).
2. Открыть проект → вкладки **Planned | In progress | Closed** по **задачам**, где пользователь назначен.

### 6.2 Ограничения

- **Нет** списка «все задачи проекта» — только **свои** назначения.
- **Нет** PM-вкладки Archive с тем же смыслом — у работника закрытые в **Closed**.
- Назначение и снятие с задачи — **только у PM** в веб-проекте.

## 7. Пошаговые сценарии тестирования

Используйте **готовые тестовые проекты, задачи и пользователей**. Для каждого сценария — **отдельный браузер / режим инкognito** или разные устройства.

### Предусловия

1. Есть проект с командой и несколькими задачами в разных статусах.
2. Есть минимум три учётки: **PM**, **Foreman**, **Worker** — все уже в команде проекта.
3. PM может открыть веб-приложение; бригадир и работник — тот же сайт, но интерфейс **Tasks** (мобильная вёрстка).

---

### Тест A — PM назначает бригадира и работника на задачу

1. Войти как **PM**.
2. Открыть проект → **Tasks** → выбрать задачу со статусом **planned**.
3. В **General** выбрать **Foreman / Brigadier** (тестовый бригадир) → **Save**.
4. Вкладка **Team** → **+ Add Team Member** → добавить тестового **Worker** → **Save**.
5. **Ожидание:** задача сохранена без ошибки; при повторном открытии lead и member на месте.

---

### Тест B — PM видит назначение в Team → Tasks (проект)

1. Оставаясь PM: проект → **Team**.
2. В строке **Worker** нажать **Tasks**.
3. **Ожидание:** открывается экран **Worker schedule (view only)** с именем работника.
4. Вкладка **Planned** — назначенная задача из теста A (статус planned).
5. **Back to team list** — возврат к таблице без перезагрузки страницы.
6. В строке задачи — метка **Team member**.

7. То же для **Foreman** → **Tasks** → та же задача с меткой **Task lead**.

---

### Тест C — PM: глобальный Team vs проектный Team

1. PM → верхнее меню **Team** → **Tasks** у того же Worker.
2. **Ожидание:** если назначения только в одном проекте — один блок с вкладками; если в нескольких — **аккордеон по проектам**.
3. PM → тот же Worker → проект → **Team** → **Tasks**.
4. **Ожидание:** только задачи **этого** проекта, без лишних проектов.

---

### Тест D — Gantt и назначение

1. PM → проект → **Tasks** → **Gantt**.
2. **Ожидание:** полоски задач по датам; клик по задаче открывает панель редактирования.
3. Убедиться, что **Foreman** и **Team** те же, что в списке Tasks.
4. PM → **Team** → **Tasks** у worker — данные **согласованы** с карточкой задачи.

---

### Тест E — смена статуса и вкладки PM

1. PM → **Tasks** → задачу из теста A перевести в **in_progress** → Save.
2. PM → **Team** → Worker → **Tasks**.
3. **Ожидание:** задача ушла из **Planned**, появилась в **Current** (Current / Upcoming).
4. PM → перевести задачу в **completed**.
5. **Ожидание:** задача в **Archive** у PM; у Worker — в **Closed** (тест F).

---

### Тест F — работник видит только свои назначения

1. Войти как **Worker** (не PM).
2. **Tasks** → **My projects** → открыть проект.
3. **Ожидание:** в **Planned** или **In progress** — задача из теста A (пока не completed).
4. Открыть карточку задачи — имя, адрес, статус, даты; **нет** кнопок назначения команды.
5. **Ожидание:** задачи, где Worker **не** в team_members и не lead — **отсутствуют**.

---

### Тест G — бригадир: My tasks vs All tasks

1. Войти как **Foreman** из теста A.
2. **Tasks** → проект → **My tasks** → **Planned** — задача, где он **lead**.
3. Переключить **All tasks**.
4. **Ожидание:** видны **все** задачи проекта, включая те, где бригадир не назначен.
5. Открыть задачу, где он **lead** → проверить возможность изменить **progress** (если UI доступен).
6. Открыть задачу, где он **не** назначен → только просмотр.

---

### Тест H — Details без лишнего

1. PM → **Team** → **Details** у любого работника.
2. **Ожидание:** профиль, профессия, языки, emergency contact.
3. **Ожидание:** **нет** блока «Current Projects» / списка проектов в этом окне.

---

### Тест I — сквозная матрица (рекомендуется для приёмки)

Подготовить **одну задачу на worker** и прогнать статусы по таблице §3.4, отмечая галочкой PM / Worker / Foreman (My) / Foreman (All).

## 8. Сообщения и уведомления (не для приёмки сейчас)

| Что | Статус |
|-----|--------|
| Чат **Foreman / PM** в слоте недельного **Schedule** | UI и контракт API описаны; проверка **отдельным** гайдом по расписанию |
| Email / SMS / push «вас назначили на задачу» | **Не готово** — нет production-рассылающего контура |
| Еженедельные digest-уведомления | В ТЗ (`docs/TZ_FIELD_VISITS_NOTIFICATIONS.md`); **не MVP** |

**При тестировании назначений:** не ожидайте письмо или push на телефон — проверяйте **только экраны** в веб и `/tasks`.

---

## 9. Чеклист если что-то не сходится

| Симптом | Что проверить |
|---------|----------------|
| Worker не видит задачу | Назначен ли в **Team** вкладке задачи; сохранена ли задача PM |
| Дубликаты строк в Team | Обновить страницу; при повторе — сообщить разработчикам (исправление dedupe) |
| PM Tasks пустой | У человека нет ни одной задачи как lead или member |
| Foreman не меняет progress | Он должен быть **task lead**, не только member |
| Gantt и Team расходятся | Gantt показывает **задачи**; Team → Tasks — **назначения по людям** |
